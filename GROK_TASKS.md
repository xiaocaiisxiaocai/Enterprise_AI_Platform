# Grok 后续任务清单

## 1. 执行背景与边界

当前 `main` 已具备 Gate F 本地确定性切片：固定单企业 Tenant、合成身份、批准快照、ACL 预过滤、统一拒答、引用、19 条回归、12 个 Golden 用例、哈希链 Trace 和 Evidence Bundle。开始前先阅读 `AGENTS.md`、`README.md`、`docs\21_首个试点用例与验收基线.md`、`docs\24_测试评测与证据追踪计划.md`，并确认工作区 clean。

本批任务不得接入 OIDC、SharePoint、数据库、向量检索、大模型、云资源或真实业务数据；不得把本地合成验证描述为生产安全、业务验收或概率性 AI Evaluation。每个任务采用“先构造失败场景，再实现修复”的顺序，完成适用验证后立即创建独立 Git 提交。未经用户明确授权，不得 push、改写历史或合并分支。

## 2. 第一批：门禁可信度（P0）

### G-01：Golden Dataset 契约回归

为评测器增加 `REG-EVAL-*` 负向回归，至少覆盖：重复 Case ID、未知主体、错误 Tenant、非法状态、空问题、重复/空白/不存在的文档 ID、期望与禁止引用重叠、回答用例无期望引用、拒答用例含期望引用、复用已有 Trace 路径。每个损坏样例必须稳定返回非零退出码，且不能生成伪造的 Passed 报告。

交付物：无外部测试包的回归执行器、清晰的 Case ID、README/测试计划同步。验收：正常 12/12 通过，全部损坏样例失败，Release 构建零警告。

### G-02：Evidence Bundle 离线验证器

新增 `scripts\Test-GateFEvidence.ps1`，只读验证证据包：schema/status、commit 与 clean 标记、快照和数据集 SHA-256、19 条回归 ID、12 条评测指标、报告与 Trace 文件名、Trace 条目数及最终哈希。验证器必须拒绝字段缺失、计数矛盾、哈希篡改、路径越界、报告/Trace 不匹配和未知 schema。

交付物：验证脚本、正反样例自测、使用说明。不得依赖 GitHub Artifact、网络或当前工作区恰好存在的未声明文件。

### G-03：证据生成的原子性与失败清理

审查 `Export-GateFEvidence.ps1` 的中途失败行为。先写入同目录临时文件，全部验证通过后再原子发布最终报告；失败时不得留下名称看似正式但内容不完整的证据包。不得删除历史证据或使用宽泛递归清理。

验收：模拟评测失败、文档失败和无效输出路径，最终正式文件保持不存在或保留上一次完整版本；成功路径仍可被 G-02 验证。

## 3. 第二批：本地安全契约（P1）

### G-04：HTTP 输入边界扩展

扩展 `REG-API-*`，覆盖空白问题、超长问题、空/未知测试身份、错误 Content-Type、畸形 JSON、未知字段、错误字段类型和并发请求。断言稳定的 4xx、无授权内容泄漏、无问题原文落入 Trace；不要为了测试改变 Production 默认拒绝测试身份的规则。

### G-05：Trace 隐私与完整性回归

增加对每条 Trace 的结构白名单验证，证明不保存问题原文、答案、文档正文或隐藏文档元数据；验证 sequence 连续、previous hash/entry hash 正确、策略与快照锚点存在。增加截断、重排、重复、删除和追加伪造记录的篡改测试。

### G-06：批准快照契约加固

补充 manifest 的严格输入边界：未知 JSON 字段、重复 Document ID、绝对路径、大小写/规范化路径冲突、空 Owner/分类/版本、重复 ACL、超大文件以及哈希格式错误。保持 fail-closed，并确保异常信息不输出文档正文或敏感路径细节。

## 4. 第三批：工程化收口（P2）

### G-07：CI 最小权限与超时

审查 `.github\workflows\docs-quality.yml`：声明只读最小权限，为 Job 设置合理超时，避免同分支过期运行浪费资源，并确保文档、PoC、evaluation、脚本变化都能触发门禁。不要引入需要仓库密钥或企业权限的步骤。

### G-08：机器可读验证摘要

让本地脚本和 CI 输出简洁一致的最终摘要，至少包括 commit、回归计数、Golden 计数、越权引用数、数据集哈希、Trace 最终哈希及限制声明。日志中预期的 4xx 不应伪装成未处理的测试失败，但必须保留断言证据。

### G-09：文档一致性与操作手册

同步 `README.md`、`AGENTS.md`、文档索引、路线图和测试计划，给出一条生成证据、一条离线验证证据的命令。检查章节编号、术语、相对链接和 Planned/Implemented 状态；不得新增与代码不一致的架构承诺。

## 5. 每项任务的完成定义

每个任务提交前必须运行适用的文档自测、Release restore/build/run、Golden 评测、Evidence 生成/验证、`git diff --check` 和 `git status --short`。提交只包含该任务文件，标题示例：`test(evaluation): 覆盖损坏 Golden Dataset`。最终汇报必须区分：已验证、仅文档规划、因权限延期；列出提交 SHA、实际命令和结果，不得只说“已完成”。

建议顺序：`G-01 → G-02 → G-03 → G-04/G-05/G-06 → G-07 → G-08 → G-09`。G-04 至 G-06 可分别提交，但不得并行修改同一回归入口而互相覆盖。
