# 04 数据库模型设计

> 状态：**Planned（目标设计，尚未实现）**
>
> 一期数据基线：**PostgreSQL + pgvector**
>
> 领域事实源：[02 DDD领域模型设计](02_DDD领域模型设计.md)、[14 DDD详细领域模型](14_DDD详细领域模型.md)

## 1. 目标与非目标

数据库目标是支持 Tenant 隔离、领域自治、不可变版本、可靠执行、知识向量检索、审计、保留删除和未来服务拆分。

一期不引入独立向量数据库，不使用单一大表承载多个领域，不依赖跨 Schema 级联删除，也不在数据库保存明文密钥、模型 Token 或外部系统密码。本文是逻辑/物理模型评审基线，不构成迁移脚本完成或性能达标证据。

## 2. 数据架构原则

1. 每个限界上下文拥有独立 PostgreSQL Schema：`identity`、`agent`、`knowledge`、`tool`、`workflow`、`governance`、`evaluation`；`platform` 仅保存无业务所有权的技术目录和计量数据。
2. 一期可共享同一 PostgreSQL 集群，但禁止模块跨 Schema 直接写入；跨 Context 仅保存 ID + Version，避免级联 FK。
3. 所有 Tenant 资源必须含 `tenant_id`，应用过滤与 PostgreSQL Row-Level Security（RLS）双重防护。
4. 发布后的 Version、PolicyDecision、AuditEvent、EvaluationResult 和事件记录追加写，不允许原地修改。
5. 外部大对象、完整模型输入输出和文件内容保存到受控对象存储；数据库只保存摘要、元数据和加密引用。
6. 所有凭据字段均为 `secret_ref`，解析凭据只发生在获准的执行边界。
7. 业务写入与 Outbox 事件在同一事务提交；消费者使用 `event_id` 幂等。

## 3. Schema 与扩展

```text
identity    Tenant、Principal、组织、角色和绑定
agent       Agent/Prompt/Policy 版本、发布、执行、步骤和 Memory
knowledge   Source、DocumentVersion、Chunk、Embedding 和 ACL
tool        ToolVersion、调用与结果状态
workflow    WorkflowVersion、Instance、Task、Approval、Timer、Signal
governance  PolicyVersion、Decision、Audit、Risk、Kill Switch
evaluation  Suite、Case、Run、Metric、Feedback
platform    模型目录、模型调用计量和跨模块技术元数据
```

一期计划启用 `pgvector`。扩展版本、备份兼容性和升级流程必须锁定在迁移清单中；启用扩展不等于向量检索性能已达标。

## 4. 通用字段与约束

### 4.1 可变聚合表

| 字段 | 类型建议 | 约束 |
|---|---|---|
| `id` | `uuid` | PK；由平台生成，不暴露业务语义 |
| `tenant_id` | `uuid` | NOT NULL；Tenant 内查询首要过滤键 |
| `status` | `text`/受控枚举 | NOT NULL；值由领域模型定义 |
| `row_version` | `bigint` | NOT NULL；乐观并发，每次变更递增 |
| `created_at` / `updated_at` | `timestamptz` | UTC、NOT NULL |
| `created_by` / `updated_by` | `uuid` | Principal 或服务主体引用 |
| `metadata` | `jsonb` | 仅放非关键扩展字段；必须有大小和 Schema 限制 |

并非所有表都允许软删除。执行、审计和评测证据按保留策略归档；用户可删除数据通过 Tombstone 和删除任务传播，不用无约束 `deleted=true` 代替。

### 4.2 不可变版本表

统一包含：`id`、`tenant_id`、父资源 ID、`version`、`content_hash`、`schema_version`、`created_at/by`、`published_at/by`。已发布版本的内容和版本元数据禁止 UPDATE/DELETE；后续弃用、撤回、过期、归档及跨资产晋级通过所属领域的追加式生命周期/Release 事件表达，并投影当前有效状态，不能回写版本内容行。状态值不是全平台共享枚举，必须由所属聚合的状态事实源定义。

### 4.3 命名与时间

- 表、字段、索引使用 `snake_case`，表名用复数；
- 时间统一 `timestamptz` 和 UTC；业务时区作为显式字段保存；
- 金额使用 `numeric` + `currency`，不使用浮点数；
- 每个聚合的状态值由该聚合唯一事实源定义；Agent、Workflow/Task/Approval 和 Tool result 不共用枚举，数据库约束与代码/契约同步生成或验证；
- JSONB 不替代核心可查询字段、FK、唯一约束和状态机。

## 5. Tenant 与 RLS

### 5.1 隔离规则

- Tenant 从已验证 OIDC Claims、客户端注册或服务绑定解析，不接受请求体自由指定；
- 数据库连接在事务开始使用受控方式设置 Tenant 上下文，连接归还池前必须清理；
- RLS Policy 对 `SELECT/INSERT/UPDATE/DELETE` 均生效，并强制 `tenant_id` 与会话 Tenant 一致；
- 平台运维访问使用单独角色、限时授权和审计，不允许应用角色绕过 RLS；
- 向量检索必须在距离排序前应用 Tenant、KnowledgeBase 和 ACL 过滤；
- 全局目录如模型能力也通过显式只读视图暴露，不与 Tenant 数据混表。

### 5.2 验收要求

至少验证跨 Tenant 主键猜测、分页游标复用、向量近邻泄漏、缓存键缺失 Tenant、后台任务 Tenant 丢失和运维角色误用。

Knowledge 有且只有一个对外有效 `lifecycle_status`，以 `document_version_lifecycle_events` 的合法事件序列投影，并以 `16_Knowledge数据治理设计.md` 为领域事实源；`promotion_stage` 从 `knowledge_release_events` 投影。投影可重建，不是第二事实源。IngestionJob 和 Chunk 只保存各自作业/索引状态，不允许写入 Published、Canary 等知识发布结论。

### 8.1 pgvector 设计

- 每个 `embedding_space` 固定模型版本、维度、距离度量和归一化规则；不同空间的分数不得直接比较。
- `VECTOR(N)` 的 `N` 由具体 Embedding Space 的物理迁移固定；模型维度变化创建新空间并双索引迁移，不原地覆盖旧向量。
- HNSW/IVFFlat、列表数、探测参数和分区方式由真实语料、召回率、延迟、写入量和内存基准测试决定；一期不得仅凭默认值宣称达标。
- 查询顺序必须在近邻候选阶段包含 `tenant_id + knowledge_base_id + ACL/classification` 约束，随后重排和二次授权过滤。
- 返回结果必须带 `chunk_id + document_version_id + locator + embedding_space_id`，支持证据复核。
- Tombstone 完成前资源不得再被检索；缓存和派生索引纳入删除任务。

## 9. Tool Schema

| 表 | 关键字段 | 关键约束/索引 |
|---|---|---|
| `tools` | `id, tenant_id, tool_key, name, owner_principal_id, status` | `(tenant_id, tool_key)` 唯一 |
| `tool_versions` | `tool_id, version, input_schema, output_schema, risk_level, side_effect_class, endpoint_ref, auth_profile_ref, timeout_ms, retry_policy, idempotency_mode, network_policy_ref, content_hash` | Published 后不可变；不保存 Secret |
| `tool_executions` | `id, tenant_id, agent_execution_id, agent_step_id, tool_version_id, principal_id, policy_decision_id, approval_task_id, step_up_evidence_ref, action_hash, idempotency_key, status, result_state, request_ref, response_ref, request_hash, response_hash, result_observed_at, resolution_evidence_ref, reconciliation_task_id, started_at, completed_at, error_code, trace_id` | `(tenant_id, tool_version_id, idempotency_key)` 条件唯一；按执行/状态索引 |

`status` 表示 Tool 调用生命周期，`result_state` 表示外部业务结果，二者不得混用。`result_state` 至少区分 `Succeeded`、`Failed`、`ResultUnknown`、`Compensated`。`ResultUnknown` 只能根据源系统查询、回调或人工对账证据解析为确定结果，更新时保留原始观察事件与 `resolution_evidence_ref`；未解析前，关联 AgentExecution 不得进入终态，也不得盲目重放非幂等调用。Approval 或 Step-up 证据必须绑定 `action_hash`，参数改变后失效。日志和数据库默认只保存脱敏摘要，完整载荷按分类存入受控对象存储。

| `audit_events` | `id, tenant_id, sequence_no, event_type, actor_id, resource_ref, action, outcome, decision_id, trace_id, correlation_id, occurred_at, payload_redacted, integrity_hash, previous_hash, classification` | 仅 INSERT；按 Tenant/时间分区；完整性链 |

该哈希链用于发现缺失、重排或内容变更，并不单独构成“不可抵赖”。如适用性评估要求抵御数据库管理员或密钥持有者篡改，须增加规范化编码、独立签名检查点、可信时间或外部锚定/WORM、密钥职责分离及可重复验证协议，并通过专项 ADR 固定信任边界。

| `model_catalog` | 供应商、模型、能力、区域和状态目录 | 无 Secret；变更版本化 |
| `model_invocations` | Execution/Step、模型版本、路由、token、费用、延迟、结果状态 | 脱敏；Tenant/时间分区 |
| `{schema}.outbox_messages` | 源模块可靠事件 | 与业务写同事务；`event_id` 唯一；追加写 |
| `{schema}.inbox_receipts` | 消费幂等与处理结果 | `(consumer, event_id)` 唯一 |
| `schema_migrations` | 每模块迁移版本和校验和 | 禁止手工漂移 |

Outbox/Inbox 采用每模块所有的表模板，避免一个共享业务表成为模块耦合点。

## 14. 索引、分区与性能

- 所有 Tenant 热路径索引以 `tenant_id` 为前导或在执行计划中证明等效；
- Execution、Task、Job、Outbox 按 `status + next_attempt_at/due_at` 建部分索引，避免全表轮询；
- Audit、ModelInvocation、ExecutionStep 等高增长表按时间并结合 Tenant 分区策略评估；
- 外键列、幂等键、Trace/Correlation、版本引用建立必要索引；
- JSONB GIN、全文索引和向量索引只在查询证据支持时创建；
- 每个关键查询保留基准数据规模、`EXPLAIN (ANALYZE, BUFFERS)` 结果和回归阈值；
- 慢查询、锁等待、膨胀、Autovacuum、连接池和向量索引内存纳入监控。

## 15. 数据生命周期与安全

1. 对每张表定义 Owner、分类、保留期、归档、删除、Legal Hold 和备份恢复要求。
2. PII、Prompt、Document、Tool 参数和模型输出遵循最小化；敏感载荷使用字段/对象级加密和独立密钥。
3. 删除流程采用 `Requested → Tombstoned → Propagating → Verified/Failed`，覆盖 PostgreSQL、pgvector、对象存储、缓存、Memory、评测副本和备份策略。
4. 审计保留与用户数据删除可能冲突时，仅保留合规所需的最小脱敏证明，并记录法律依据。
5. 恢复演练必须证明 Tenant 映射、RLS、版本快照、Audit 顺序和向量索引均可恢复；备份成功日志不等于恢复成功。

## 16. 迁移策略

- 每个模块独立迁移序列，迁移文件不可改写；
- 采用 Expand → Backfill → Dual-read/验证 → Contract，破坏性删除不得与应用切换同一步；
- 大表 DDL、向量重建和分区变更需评估锁、磁盘、复制延迟和回滚；
- 数据回填具备 Tenant 范围、检查点、幂等键、限速和校验报告；
- Schema 与应用版本兼容窗口、回滚限制和 Feature Flag 写入发布方案；
- 生产手工 SQL 必须走审批、演练和审计流程。

## 17. 失败路径

| 失败 | 目标行为 |
|---|---|
| 事务提交失败 | 业务与 Outbox 同时回滚，不返回成功 |
| 并发版本冲突 | 返回明确冲突，由调用方重读后决定，不覆盖 |
| 向量索引不可用 | 在容量允许且策略批准时降级为受 ACL 过滤的精确/替代检索，否则明确不可用 |
| 删除传播部分失败 | 保持 Tombstone，禁止召回，重试并告警，直至验证完成 |
| Audit 完整性校验失败 | 生成高严重度 RiskEvent，停止受影响高风险操作并保全证据 |
| 数据库主节点故障 | 按运维设计切换；未确认写入通过幂等键对账 |
| 迁移中断 | 从迁移检查点恢复或回滚兼容应用，不手工标记完成 |

## 18. 评审与验收点

- [ ] 所有 Tenant 表均有 `tenant_id`、RLS 和跨 Tenant 隔离测试。
- [ ] 每个 AgentExecution 五类版本引用均 NOT NULL，`snapshot_hash` 可复核。
- [ ] Published Version、PolicyDecision、AuditEvent、EvaluationResult 不可原地更新。
- [ ] ToolExecution 的策略决定、ActionHash、幂等键和 `ResultUnknown` 可追踪。
- [ ] pgvector 查询在 ACL 前置过滤下完成召回率、延迟和资源基准测试。
- [ ] Outbox 原子性、消费者幂等、乱序和重复投递通过故障测试。
- [ ] 删除请求可证明传播到 Chunk、Embedding、对象、Memory、缓存和评测副本。
- [ ] 备份已完成实际恢复演练，并验证 RLS、审计完整性和向量索引。
- [ ] 数据类型、约束、索引和迁移 DDL 与本文一致；差异有 Accepted ADR。
