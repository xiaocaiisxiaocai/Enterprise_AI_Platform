# 13 开发路线规划

> 状态：Planned（目标设计，尚未实现） ｜ 规划方式：以业务结果和退出门禁推进，不以功能清单或固定日期宣称完成

## 1. 路线原则

1. **可信基线前置**：身份、租户、默认拒绝、Secret、审计、Trace、评测和成本标签从 Phase 0 建立，后续只提升成熟度。
2. **模块化单体起步**：先保持清晰逻辑边界，不为“最终架构”预建微服务；拆分必须有容量、隔离、团队或合规证据。
3. **只读先行、副作用后置**：Knowledge 和 Agent 先提供只读价值；任何副作用 Tool 与审批、幂等、补偿、紧急禁用同时交付。
4. **每阶段可上线也可回滚**：阶段完成以业务试点、质量、安全、SLO、成本和恢复证据为准，不以 Demo 演示为准。
5. **候选不等于依赖**：LangGraph、Temporal、Kafka、专用向量库等须经 ADR/PoC；一期基线为模块化单体、PostgreSQL + pgvector、Outbox + Worker。
6. **规划技术不伪装成现状**：文档中的 Proposed/Candidate 必须在代码、配置和验收证据完成后才能标为 Implemented。

## 2. 全阶段 Definition of Done

每个阶段必须同时满足：

- 有已批准的业务用例、目标用户、Owner、范围和成功指标；
- 威胁模型、数据分类、权限矩阵和租户隔离测试已更新；
- 单元、契约、集成、端到端、回归和适用的对抗测试通过；
- Candidate 按 `Evaluating → Reviewing（含安全门禁）→ Canary → Published` 推进，并完成回滚演练；
- 每个生产用例登记非空 SLO、错误预算、Dashboard、告警、Runbook 和成本预算；
- Schema/数据/配置迁移向后兼容，备份与恢复按阶段风险演练；
- 文档、ADR、API/事件 Schema、运维和用户说明与实现一致；
- 未解决风险有 Owner、补偿控制和到期日，P0 风险不得豁免上线。

## 3. Phase 0：可信工程基座

### 3.1 业务目标

选定一个高价值、低副作用的企业知识试点，建立可重复交付和可信度量能力。试点范围、数据、用例和退出规则以 `21_首个试点用例与验收基线.md` 的批准版本为准；该阶段不承诺 Agent 自动执行业务动作。

### 3.2 分步交付与停止规则

Phase 0 不作为一个“大爆炸”阶段交付，按证据顺序推进：

| 步骤 | 最小产物 | 退出条件 |
|---|---|---|
| 0A 决策就绪 | 批准的业务问题、Gate F 数据切片、五帽子最小 RACI、文档自动检查 | P0 范围/责任 Open 项关闭；自动检查可阻断已知坏例 |
| 0B 架构 Spike | ADR-003、ADR-004、ADR-011 的独立最小实验与原始结果 | 每项按 `20_ADR技术决策.md` 的 PoC 矩阵单独 Pass/Fail/Blocked，不打包判定 |
| 0C 最小纵向切片 | 测试身份 → Tenant → Policy Port → 权限过滤检索 → 引用/拒答 → Trace | 两个合成 Tenant 无越权；故障显式化；结果可重复 |
| 0D 受控试点准备 | 目标 IdP/数据源、评测集、成本、支持和回滚 | 达到 `21_首个试点用例与验收基线.md` 的 Gate P 入口条件 |

每一步都应设置明确时间盒和停止条件，但具体天数由团队容量批准，不在没有人员与环境信息时虚构。Spike 失败是有效证据；失败后应更新 ADR 或范围，不得继续堆叠框架和基础设施掩盖结论。

### 3.3 目标交付范围

- 规划中的 Monorepo、模块边界、统一配置、版本和本地容器化开发环境；
- CI：格式/静态检查、测试、Secret/依赖/镜像扫描、SBOM、签名 Artifact；
- CD 基础：环境晋级、迁移、Canary、回滚和不可变版本清单；
- 企业 IdP 接入、User/Agent/Workload 身份、tenant 上下文和最小 RBAC+ABAC；
- 中央 PDP 合约、各入口 PEP、默认拒绝、Secret Broker 与追加式、可校验的防篡改审计；
- Trace/Metrics/Logs、版本标签、成本台账和 Eval Harness；
- PostgreSQL + pgvector、对象存储、Transactional Outbox + Worker 基线；
- 威胁模型、数据分类、备份/恢复、事件响应和首批 ADR。

上述列表是 Phase 0 完整目标范围，不代表必须在 0A 或 0B 一次建成；每项必须归入明确步骤和 Owner。

### 3.4 退出门禁

空环境可重复部署；跨租户测试零泄漏；PDP 故障 fail closed；审计/Trace 可串联请求；评测能阻断一个故意引入的回归；数据库迁移与恢复演练通过；试点用例、绝对质量/延迟/成本阈值已由业务 Owner 批准。验证规格和 Evidence 按 `24_测试评测与证据追踪计划.md` 登记。

## 4. Phase 1：权限感知 Knowledge Assistant

### 4.1 业务目标

让获授权用户基于企业文件获得可追溯回答，并在证据不足时可靠拒答。

### 4.2 交付范围

- File System Connector、增量/幂等摄取、恶意文件隔离和删除传播；
- PDF/DOCX/XLSX/PPTX/Markdown/图片解析及 OCR/Layout 质量门禁；
- Document/Chunk 血缘、源 ACL 投影、版本、发布和回滚；
- PostgreSQL/pgvector + Lexical Provider 的权限感知混合检索；
- claim-evidence 引用、无答案、冲突/过期知识处理和用户反馈；
- RAG Golden/Regression/Adversarial 数据集，以及 Recall@k、引用精确率、Groundedness、延迟和成本门禁。

### 4.3 退出门禁

权限预过滤、ACL 变化和删除端到端生效；跨租户/越权召回为零；解析失败进入复核而非静默发布；引用可定位源版本和证据范围；关闭 `TBD-EVAL-001` 后，质量达到已批准且版本化的 EvalPolicy 门槛；试点用户验收并确认收益基线。`11_Evaluation_Monitoring_Cost设计.md` 中标记为示例的数字不得替代该批准过程。

## 5. Phase 2：受限 Agent Runtime

### 5.1 业务目标

在只读或无业务副作用场景中，让 Agent 完成多步检索、分析和结构化建议，并能解释失败和恢复。

### 5.2 交付范围

- Agent/Prompt/模型版本 Registry、会话预算、超时、取消和并发限制；
- Planner、结构化输出、Reviewer/Guardrail，以及明确的最大步骤和停止条件；
- Memory 分级、用途/保留/删除和防投毒；长期 Memory 写入先进入 Candidate；
- 模型路由、供应商策略、Fallback、Trace 与成本归集；
- 轨迹、任务成功、拒答、恢复、Prompt Injection 和数据外泄评测。

LangGraph 只作为有状态 Agent 机制候选；是否采用由 ADR 和故障恢复 PoC 决定。

### 5.3 退出门禁

Agent 不能绕过 Knowledge/PDP；步骤、Token 和成本有硬上限；模型/Prompt 变更可回归和回滚；恶意检索内容不能取得 Tool 权限；只读试点达到任务成功、质量、延迟和成本目标。

## 6. Phase 3：受治理 Tool、SDK 与 MCP

### 6.1 业务目标

先让 Agent 安全查询业务系统，再有限开放可逆、低影响动作；不存在“先接 Tool、以后补审批”的阶段。

### 6.2 交付范围

- 完整 Tool Registry、JSON Schema、风险/副作用、Owner、版本和弃用；
- Tool Gateway PEP、短期凭据、Schema/DLP、限流、超时、幂等和全路径审计；
- 只读 Tool 优先；第一个副作用 Tool 上线前同步交付最小 Workflow/HITL；
- Python/C# AI SDK、Mock、契约测试、Trace 和统一错误；
- MCP Host/Client/Server，稳定规范版本固定为 `2025-11-25`，支持显式批准的能力和安全传输；
- Skill/MCP Server 供应链：来源、签名、SBOM、隔离、网络白名单、紧急禁用和回滚；
- A2A 仅定义身份、任务、Artifact 和状态兼容边界，不承诺一期实现。

OpenClaw 只借鉴 Gateway、渠道、会话、沙箱和技能治理，不采用其单用户信任模型；Hermes 类自学习 Skill 必须执行统一的 `Candidate → Evaluating → Reviewing → Canary → Published` 生命周期，并支持 `Rejected / Quarantined / RolledBack / Superseded`。

### 6.3 退出门禁

未授权副作用为零；Deny、审批、过期、撤销、换参、重复请求、超时未知和紧急禁用均通过测试；本地/远程 MCP 越权、SSRF 和 token passthrough 被阻断；每个 Tool 有 Owner、SLO、成本和回滚证据。

## 7. Phase 4：Workflow Automation 与完整 HITL

### 7.1 业务目标

将已验证的单步能力组合成长流程，在中断、重试、人工介入和部分失败时保持业务可恢复。

### 7.2 交付范围

- Versioned WorkflowDefinition、Instance/Task/Approval 状态机和事件历史；
- 审批队列、增强认证、职责分离、委托/升级、截止和执行快照；
- Pause/Resume、Timer、幂等重试、取消、对账、Saga/补偿和人工接管；
- Transactional Outbox/Inbox、Schema 版本、死信和受审计重放；
- Workflow/审批等待/补偿 SLO、故障恢复和业务级评测。

Temporal 只作为长流程与持久执行候选；必须在一期状态机无法满足经测量的恢复、规模或运维需求后再评审。

### 7.3 退出门禁

故障后历史可恢复；重复事件不产生重复副作用；审批与执行快照一致；补偿失败可升级人工；关键工作流完成灾难恢复、并发和业务验收演练。

## 8. Phase 5：企业级规模化与运营成熟

该阶段不是第一次引入 Governance、Monitoring、Cost 或 Deployment，而是将已有能力提升到企业规模：

- SaaS/私有化/混合部署产品化，环境、租户、地域和共享责任清晰；
- HA、容量、性能、自动扩缩容、RPO/RTO、异地备份和恢复演练；
- 合规适用性、证据自动化、审计运营、红队与供应链治理；
- FinOps、Showback/Chargeback、用例组合 ROI 和供应商治理；
- 基于指标评估是否拆分微服务、引入 Kafka、专用向量库或外部 Workflow Engine；
- 用户培训、运营流程、支持等级和变更管理。

退出门禁由正式生产准备度评审决定：SLO/错误预算达标，容量与灾备通过，安全/合规无未关闭 P0，单位成功成本和业务收益可验证，回滚及事件响应演练完成。

## 9. 依赖与晋级规则

阶段是能力依赖，不要求所有工作完全串行；但不得跨越以下硬门槛：

| 计划能力 | 前置门槛 |
|---|---|
| RAG 试点 | 身份/租户、ACL 摄取、审计、Eval Harness |
| Agent 试点 | RAG 质量门禁、预算/Trace、只读边界 |
| 只读 Tool | Registry、PDP/PEP、短期凭据、沙箱和审计 |
| 副作用 Tool | Workflow/HITL、幂等、TOCTOU 防护、补偿/恢复 |
| 自动 Workflow | 状态恢复、事件去重、审批与业务对账 |
| 规模化生产 | SLO、容量、DR、合规证据、FinOps 与运营 Owner |

## 10. 组合风险与控制

- **过早微服务**：一期保持模块化单体；用指标和 ADR 决定拆分。
- **过度 Agent 化**：确定性规则和流程保留在 Workflow/业务系统；Agent 只处理确有不确定性的任务。
- **无治理上线**：治理、审计、评测从 Phase 0 存在，任何阶段均不可旁路。
- **平台先于场景**：每阶段绑定真实用户、业务基线和收益假设，未被采用的通用能力不扩建。
- **自动改进失控**：模型、Prompt、知识、Policy 和 Skill 只能生成 Candidate，经 `promotion_stage` 的 Evaluating、Reviewing（含安全门禁）和 Canary 后发布；该晋级覆盖层不得替代各领域状态机。
- **供应商/框架锁定**：协议和领域合约在平台侧稳定，候选组件通过 Adapter、兼容测试和退出方案隔离。
- **成本随规模失控**：预算、配额和单位成功成本从试点开始测量，错误预算或成本预算耗尽时暂停非修复发布。

## 11. 路线治理

每个 Phase 维护范围、Owner、依赖、决策、风险、验收证据和发布记录。季度或重大事故后重审路线，但不能修改历史验收结果；范围变化通过 ADR/变更记录说明原因和影响。

阶段 Accountable、职责分离、人工队列容量和升级路径以 `22_组织责任与RACI运营模型.md` 的批准映射为准；抽象 Owner 未映射真实主体时不得进入对应发布门禁。

进度只使用 `NotStarted | InProgress | Blocked | Validated | Released`。没有运行证据只能标为 `InProgress`，CI 成功不能替代预生产/生产验证，规划文档完成也不能标记平台能力已交付。
