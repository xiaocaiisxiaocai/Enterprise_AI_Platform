# 24 测试、评测与证据追踪计划

> 状态：**PartiallyImplemented（文档门禁、Gate F 本地回归与确定性评测已实现，正式产品验证尚未执行）** ｜ 责任域：Evaluation / Quality Engineering ｜ 适用阶段：Phase 0 起 ｜ 原则：需求可判定、风险可复现、证据不可伪造

## 1. 目的与边界

本文定义从需求、威胁、契约到测试、评测、演练和发布证据的统一闭环。仓库已有文档门禁、Gate F 最小 API、确定性回归执行器、12 用例 Golden 契约、本地哈希链 Trace 和机器可读 Evidence Bundle。它们证明本地固定 Tenant、部门 ACL、拒答、引用、Trace 失败关闭与 HTTP 输入边界可以重复验证；真实 IdP、SharePoint、pgvector、模型、正式环境和业务数据仍不存在，因此正式 Verification/Evidence 继续为 Planned/NotRun，PoC 通过不等于试点或平台实现通过。

测试验证确定性功能、权限和恢复；Evaluation 衡量概率性质量、安全、成本与业务结果；Review/Exercise 验证人工决策和运营恢复。三者不能互相替代。

## 2. 核心原则

1. **风险优先**：跨安全域/资源 ACL、未授权副作用、密钥泄露和审计缺失先于覆盖率数字。
2. **原子追踪**：一个 Verification 对应一个可判定需求或攻击路径，禁止用“整体验收通过”掩盖失败。
3. **冻结被测对象**：记录代码、制品、配置、Prompt、模型、数据集、Policy、Tool、Schema 和环境版本。
4. **确定性优先**：Schema、授权、计算、状态转换和引用存在性使用确定性断言；模型 Judge 不能替代硬规则。
5. **独立验证**：关键门禁不能仅由生成同一实现/Prompt 的模型或人员评审。
6. **诚实证据**：失败、跳过、环境阻塞和样本不足均显式记录；设计、截图和日志片段不能冒充完整运行证据。
7. **可重复与可删除**：数据集有来源、版本、分类、保留和删除传播；敏感用例不上传未批准外部服务。

## 3. Verification 与 Evidence 标识

| 类型 | ID 模式 | 示例 |
|---|---|---|
| 功能/契约测试 | `VER-FR-<DOMAIN>-NNN-NN` | `VER-FR-IAM-001-01` |
| 非功能验证 | `VER-NFR-<DOMAIN>-NNN-NN` | `VER-NFR-REL-001-01` |
| 威胁验证 | `VER-THR-<DOMAIN>-NNN` | `VER-THR-TEN-001` |
| AI 评测运行 | `EVAL-<USECASE>-<SEQ>` | `EVAL-KNOW-001` |
| 演练 | `EXR-<TYPE>-<SEQ>` | `EXR-RESTORE-001` |
| 业务验收 | `UAT-<PILOT>-<SEQ>` | `UAT-PILOT-KNOW-001` |
| 证据包 | `EVD-<VERIFICATION>-<RUN>` | `EVD-VER-THR-TEN-001-001` |

ID 创建后不得复用。重跑生成新 Run/Evidence ID，并关联前一运行；不得覆盖失败结果。

## 4. 测试与评测层次

| 层次 | 目标 | 典型对象 | 门禁阶段 |
|---|---|---|---|
| 文档/静态 | 链接、围栏、Schema、Secret、许可证和术语一致 | Markdown、OpenAPI、JSON/YAML、SBOM | 每次变更 |
| Unit / Property | 领域不变量、边界、纯函数和生成式属性 | 状态转换、预算、规范化、ACL 交集 | Phase 0 起 |
| Architecture | 防止跨模块内部引用、绕过 PEP 和共享数据所有权 | 模块依赖、Schema/Port 边界 | Phase 0 起 |
| Contract | Provider/Consumer、兼容、错误和事件 Schema | REST、MCP、Model Gateway、Tool、Outbox | 接口变更 |
| Integration | 数据库、对象存储、IdP、PDP、Connector、模型/Tool Adapter | 模块与外部依赖 | 环境晋级前 |
| End-to-End | 真实用户流程与权限传播 | 登录→检索→引用/拒答→审计→反馈/删除 | 试点发布前 |
| AI Evaluation | 质量、拒答、安全、成本和漂移 | RAG、Agent、Model Route、Reviewer | Candidate/Canary |
| Security/Abuse | 攻击路径和控制强制点 | Tenant、Prompt Injection、SSRF、供应链 | 发布阻断 |
| Resilience/Fault | 重试、对账、恢复、补偿和降级 | Worker、模型、PDP、DB、队列、审计出口 | 发布/演练 |
| Performance/Capacity | 延迟、吞吐、饱和点、积压和成本 | API、检索、索引、模型、人工队列 | 容量门禁 |
| DR/Operations Exercise | RTO/RPO、备份恢复、Runbook 和指挥 | 数据、环境、密钥、删除账本 | 生产前/复审 |
| UAT/Value | 用户任务和业务收益是否达标 | 首个试点及对照基线 | Go/Hold/Stop |

覆盖率只用于发现未测试代码，不作为质量充分性的证明。高风险路径即使代码很少，也必须有独立验证。

## 5. 需求—验证—证据账本

SRS 第 19.3 节定义最小字段，实施时使用机器可查询账本：

| 字段 | 规则 |
|---|---|
| Trace ID | 唯一且稳定；一行一个原子需求/威胁 |
| Requirement / Threat | 精确 FR/NFR/UC/THR ID，不使用范围缩写 |
| Design Location | 文件 + 章节 + 契约/Schema 版本 |
| Verification ID | 测试/评测/演练规格，不等于一次运行 |
| Method | Test / Eval / Review / Exercise / Inspection |
| Deterministic Criterion | 可明确判断 Pass/Fail，概率指标引用批准 EvalPolicy |
| Evidence ID | 实际运行后填写；未运行=`—（未执行）` |
| Owner / Approver | 唯一 A 与独立批准角色 |
| Status | Planned / Ready / Running / Passed / Failed / Blocked / Waived |

`Waived` 不适用于跨安全域/资源 ACL 泄漏、未授权副作用、密钥泄露和必要审计缺失。其他豁免必须记录风险、补偿控制、批准人、到期日和复验条件。

## 6. 测试环境与数据治理

环境至少区分本地/CI、集成、预生产和生产；身份、Secret、网络、数据和供应商账号不得跨环境复用。一期固定企业 Tenant，跨部门/资源 ACL 测试使用专用合成主体和数据；非配置 Tenant 必须默认拒绝。未来引入多个独立治理域时，再增加跨租户测试，不以真实业务数据制造越权场景。

测试数据要求：

- 明确来源、Owner、用途、分类、许可证/同意和有效期；
- 优先合成或脱敏；需要真实样本时最小化、审批、加密并限制访问；
- 保留正常、异常、边界、历史缺陷和对抗样本；
- 训练/调优、开发、门禁和最终盲测集合分离，检测数据泄漏；
- 数据删除、撤权和 Legal Hold 传播到评测副本、缓存和导出；
- 每个 DatasetVersion 固定内容摘要、Schema、采样方法和变更说明。

## 7. AI Evaluation 设计

### 7.1 数据集组合

| 数据集 | 目的 | 更新规则 |
|---|---|---|
| Golden | 代表核心业务任务与专家期望 | Owner 审核、版本化；变更触发基线重算 |
| Regression | 固化历史缺陷、事故和高频失败 | 缺陷关闭前必须加入；不得删除失败历史 |
| Adversarial | Prompt Injection、越权、投毒、外泄和滥用 | Security Owner 维护；敏感样本受控 |
| Edge/Unknown | 无答案、冲突、过期、低质量和超长输入 | 验证拒答、澄清和降级 |
| Canary/Online | 真实分布与漂移信号 | 最小采样、脱敏、用途批准；不直接训练 |
| Human Calibration | 校准 Judge 与人工一致性 | 双人/仲裁、盲评、定期复核 |

### 7.2 评测器顺序

1. 确定性：Schema、引用存在/权限、结果状态、成本与延迟；
2. 规则/程序：文本重叠、检索指标、敏感字段、Policy 正反例；
3. 人工：业务正确性、可用性、冲突和高影响判断；
4. 模型 Judge：适用于规模化语义评分，但必须固定模型/Prompt/温度、校准并报告相关性风险。

Judge 与被测对象使用同一模型或供应商时标记非独立；关键门禁结合确定性规则、不同模型/供应商或人工抽检。不得把模型自评“安全”作为高风险控制证据。

### 7.3 统计要求

每个 EvalPolicy 登记绝对阈值、允许回归、样本量、置信要求、分组维度、缺失处理和多重比较策略。样本不足时状态为 `Blocked`、`reason_code=InsufficientEvidence`，不能按通过处理。

同时报告总体和关键切片：Tenant/组织、语言、文档类型、风险、数据分类、问题长度、模型 Route 和失败类型。只看平均值可能掩盖小群体、长尾或高风险退化。

## 9. 阶段质量门禁

| Phase | 必须通过的最小证据 |

| Phase 0 | 可重复环境；身份/Tenant/PDP fail-closed；审计/Trace；迁移/恢复；安全与评测能阻断故意回归 |

## 10. CI/CD 证据流水线

### 10.1 当前已实现的文档门禁

每次文档变更运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\Validate-Docs.ps1 -SelfTest
```

脚本先用故意损坏的 JSON 和相对链接执行回归自测，证明校验器能发现已知坏例；随后检查仓库文档的单一 H1、标题跳级、代码围栏、相对 Markdown 链接、JSON 代码块、版本字段、TBD 登记和 FR/NFR/KPI 引用。GitHub Actions 在 push 与 pull request 上执行相同命令。

该门禁不检查外部 URL 内容、Mermaid 语义、自然语言事实正确性或产品行为；这些仍需要专用工具和独立评审。

### 10.3 产品流水线目标

除 Gate F 最小切片外，下列产品阶段仍为规划，不能声称已运行：

1. **Static**：格式、lint、Secret、许可证、依赖/镜像、SBOM、OpenAPI/Schema、文档链接；

## 12. 失败、阻塞与豁免

- **Failed**：断言或门禁未满足，必须修复或作出 Stop/Rollback；
- **Blocked**：环境、数据、批准或外部依赖缺失，不等于 Passed；
- **Skipped**：规格明确不适用并记录依据；不能以时间或不稳定为由跳过 P0；
- **Waived**：风险接受人批准的限时例外，包含补偿控制、范围、到期日和复验；
- **Flaky**：先隔离根因并计入质量债；不得反复重跑直到偶然通过后隐藏失败。

跨安全域/资源 ACL 泄漏、未授权副作用、密钥泄露和必要审计缺失不可豁免。模型概率性失败不能用“偶现”取消记录，应报告频率、切片和置信范围。

## 13. 回归、漂移与复验触发

以下变化触发影响范围评估与回归：代码/配置、Prompt、模型/别名、Retriever/Embedding、数据源/ACL、Policy、Tool/Skill/MCP、Schema、基础设施、供应商条款/区域/价格以及历史事故。

触发后重新运行受影响的 deterministic tests、Golden/Regression/Adversarial Eval、容量或演练；不能只运行新增测试。线上漂移、成本异常、用户申诉和人工接管进入缺陷/Candidate，不直接修改生产资产。

## 14. 独立审查

关键安全、模型风险和业务价值证据由非唯一实现者复核。审查者检查规格是否能发现缺陷、数据是否泄漏、指标是否可操纵、失败是否被排除、环境是否代表目标负载，以及证据哈希/版本是否匹配。

AI 辅助生成测试时，必须增加人工或独立方法检查模型盲点；同一模型编写实现和评测器时标记相关性风险，并为关键路径补充确定性断言、变异/故障注入或独立红队。
